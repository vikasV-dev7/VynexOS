# Version 0.3.0: Runtime Extension Framework

## Overview
VynexOS v0.3.0 transforms the system from a statically composed runtime into a dynamically extensible platform. It introduces a mathematically verifiable plugin architecture with strict lifecycle management, safe cross-boundary dependency injection, and OS-agnostic library loading.

## Architecture Components

### 1. Library Loading Abstraction (`ILibraryLoader`)
Platform-specific dynamic link library APIs (`LoadLibrary` for Windows, `dlopen` for POSIX) are hidden behind `ILibraryLoader`. The system builds the correct loader based on CMake detection without polluting core components with `#ifdef` macros.

### 2. Dependency Injection (`PluginContext`)
Plugins operate strictly within a sandbox. The `VynexPluginContext` struct exposes exactly four capabilities to the C-ABI:
- `log()`
- `publish_event()`
- `schedule_task()`
- `get_configuration()`
Plugins cannot directly access the `PluginManager`, `CompositionRoot`, or any hardware abstractions.

### 3. Centralized Context Creation (`PluginContextFactory`)
Context instantiation is decoupled from the `PluginManager`. A central factory securely creates C++ contexts and bridges them into the C-ABI structs.

### 4. Strict Lifecycle Management (`LoadedPlugin`)
The `LoadedPlugin` class wraps the raw `ISharedLibrary` handle and enforces a deterministic state machine:
**Startup Phase:**
1. Loaded
2. ABI Validation
3. Created
4. Initialized
5. Services Registered
6. Started
7. Running

**Shutdown Phase:**
1. Stopped
2. Shutdown
3. Destroyed
4. Unloaded

Any failure during the startup phase immediately triggers the shutdown phase and safely tears down the partial state.

### 5. Verification & Test Harness
An extensive test suite (`test_plugin_manager`) leverages multiple dynamic mock libraries to verify positive load paths and reject invalid ABIs, version mismatches, and initialization failures without leaking resources.
