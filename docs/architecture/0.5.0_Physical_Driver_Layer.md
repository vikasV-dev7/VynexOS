# VynexOS 0.5.0 Physical Driver Layer (SDL2/OpenGL)

## Abstract
This document outlines the architectural integration of physical display, input, and clock synchronization via SDL2, replacing the earlier memory-only mock subsystems.

## 1. Backend Abstraction Rationale
The integration of SDL2 adheres strictly to a boundary-based **PIMPL (Pointer to Implementation)** idiom.
*   **Why?**: VynexOS enforces a strict compilation firewall. If `vynexos::desktop` headers include `<SDL.h>`, then every downstream application or driver must also include SDL's massive dependency tree, dragging in macros (`#define main SDL_main`) and polluting the global namespace.
*   **How?**: `SDL2DisplayBackend` manages an opaque `std::unique_ptr<SDLContext>`. The `SDLContext` struct is defined entirely inside the `.cpp` translation unit, isolating all physical GPU API calls and dependencies behind a stable C++ ABI.

## 2. RAII Lifecycle and Resource Ownership
The `SDLContext` controls the structural sequence of GPU contexts.
- **Acquisition**: Triggered in `initialize()`. Instantiates `SDL_Window` -> `SDL_Renderer` -> `SDL_Texture`.
- **Destruction**: Destructor executes reverse teardown (`SDL_DestroyTexture` -> `SDL_DestroyRenderer` -> `SDL_DestroyWindow` -> `SDL_QuitSubSystem`).
- **Safety Guarantee**: If VynexOS crashes or triggers an emergency abort, stack-unwinding ensures standard C++ destructors immediately release VRAM preventing stale driver locks.

## 3. Thread Ownership
Hardware rendering on many operating systems (specifically macOS/Windows) demands that the Window Manager and Event Pump exist strictly on the **Main Thread**.
`CompositionRoot` deliberately reserves the `run()` blocking loop on the primary thread for this exact reason, while deferring actual OS logic to the asynchronous `TaskScheduler` background thread pool.

## 4. Failure Handling Strategy
All HAL layer initializations return modern `std::expected<void, DisplayError>`.
*   If `SDL_CreateWindow` fails (e.g. no physical display in a server rack), the error propagates to `CompositionRoot`.
*   The system gracefully drops back to headless logs without a segfault, isolating UI rendering from OS kernel functionality.

## 5. Portability & Future Rendering Backend Strategy
SDL2 currently maps `vynex_desktop` commands to OpenGL, Direct3D, or Metal based on the host OS.
However, the design relies solely on the `hal::FrameBuffer` which is an array of `ARGB8888` pixels.
This means VynexOS is fully decoupled from SDL2. In future iterations (Version 1.0+), we can hot-swap `SDL2DisplayBackend` for:
- A pure **VulkanBackend** for extreme low-overhead rendering.
- A **Wayland/DRM Backend** for running directly on Linux without an X11 server.
- A **WebAssembly Canvas Backend** for running VynexOS entirely within a browser.

## 6. Known Limitations
- The system currently enforces a fixed 60 FPS update via `FrameClock`.
- Moving windows heavily across multi-monitor setups may cause minor vsync tearing due to software buffer copying (`SDL_UpdateTexture`) rather than direct GPU compositing. This will be addressed when moving to the eventual hardware-composited Vynex UI layer.
