# Documentation Contribution Guide

This document defines the long-term maintenance rules for the VynexOS Obsidian Knowledge Base.
The documentation architecture is considered **Production Ready** and **Stable Infrastructure**. Future changes must be incremental and driven strictly by architectural evolution rather than structural reorganization.

## Documentation Philosophy
Documentation is treated as a first-class engineering deliverable. It is the canonical source of truth for architecture and design decisions. No task is considered complete until both the implementation and its corresponding documentation meet production quality.

## Information Architecture Principles
The Knowledge Base must remain a directed, layered engineering graph:
- **Logical Grouping**: Every new document must belong to exactly one logical engineering cluster.
- **Top-Down Routing**: Every document must be reachable from `Home.md` through the hub hierarchy.
- **No Strict Orphans**: Completely isolated pages are strictly prohibited.

## Hub-Based Navigation Strategy
Top-level navigation is managed via **Architecture Hubs** (e.g., `Core`, `HAL`, `Desktop`).
These hubs isolate subsystem dependencies and provide domain-specific routing. Top-level hubs may **only** be introduced following a formal architecture review. 

## Cross-Reference Rules
- Add cross-links **only** where genuine engineering relationships exist (e.g., `CompositionRoot` ↔ `EventBus`).
- Do **not** manufacture relationships solely to increase graph connectivity or artificial density.

## Architecture Page Standards
Manual architecture pages must document:
- Responsibilities and scope of the subsystem.
- Genuine dependencies on other components.
- The rationale behind structural decisions (linking to ADRs where necessary).
- Links to auto-generated Interface and Service API pages.

## Interface & Service Documentation Standards
- Public APIs (`Interfaces` and `Services`) must be thoroughly documented in the C++ source code using Doxygen-style comments (e.g., `@Purpose`, `@ThreadSafety`, `@Ownership`).
- The **Documentation Platform** will auto-generate the markdown files for these APIs. 
- Manually curated sections in these files must be placed beneath the `<!-- AUTO-GENERATED END -->` tag.

## Testing Documentation Standards
Testing architecture must be documented alongside the component it verifies. The `Testing` hub acts as the primary index for integration workflows, build-time verifications, and runtime dynamic analysis.

## Diagram Standards
Mermaid diagrams must be used for architectural topologies.
- Ensure diagrams are explicitly synchronized with the current subsystem terminology.
- Avoid using Wiki-links inside Mermaid blocks, as this breaks static site renderers.

## Naming & Wiki-Link Conventions
- Wiki-links must match the target file exactly (e.g., `[[Event Bus]]`).
- File names must use Title Case for documents and hubs. Auto-generated files will use the exact C++ symbol name (e.g., `[[IEventBus]]`).

## Markdown Standards
Use standard GitHub Flavored Markdown (GFM). Leverage semantic headings (`#`, `##`, `###`) to ensure the document outline is natively readable by both engineers and automation tools.

## Generated vs Curated Documentation
- **Generated**: Files in `docs/Interfaces/` and `docs/Services/` are generated by the pipeline. Never overwrite the top sections bounded by generation tags.
- **Curated**: Files like `Core.md`, `Event Bus.md`, and this guide are manually curated.

## Documentation Review Checklist
Before merging a documentation PR, verify:
- [ ] Document belongs to a logical parent cluster.
- [ ] Document is reachable from `Home.md`.
- [ ] No orphaned links have been created.
- [ ] Cross-references reflect genuine architectural coupling.
- [ ] Mermaid diagrams reflect the current C++ state.

## Contribution Workflow
1. Author documentation changes locally.
2. If modifying C++ APIs, run the generator: `python tools/docs/generate_docs.py --generate`.
3. Validate graph integrity: `python tools/docs/generate_docs.py --validate`.
4. Commit changes with a reference to the related engineering task.

## Definition of Done for Documentation
Documentation is "Done" when:
- It accurately describes the deployed engineering reality.
- It passes the local Validation toolchain with 0 errors.
- It fulfills all requirements in the Review Checklist above.
